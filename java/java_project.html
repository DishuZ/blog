# 后端环境搭建

资料中准备了初始工程文件夹sky-take-out，后端工程基于 Maven 进行项目构建，并且进行分模块开发。

  1. sky-take-out Maven 父工程，统一管理依赖版本，聚合其他子模块；
  2. sky-common 子模块，存放公共类，例如：工具类、常量类、异常类等；
  3. sky-pojo 子模块，存放实体类、VO、DTO 等；
    1. Entity 实体，通常和数据库中的表对应
    2. DTO 数据传输对象，通常用于程序中各层之间传递数据
    3. VO 视图对象，为前端展示数据提供的对象
    4. POJO 普通 Java 对象，只有属性和对应的 getter 和 setter
  4. sky-server 子模块，后端服务，存放配置文件、Controller、Service、Mapper 等

TIP

Q：为什么要准备初始工程，而不是从零开发？ A：在真正进入到公司工作时，一般情况下是不会让你从零构建一个项目，都是基于已有的项目进行二次开发，这里模拟的就是这种业务场景。

有很多远程仓库的选择，这里选的是 码云Gitee：https://gitee.com/ (opens new window) 。在登录之后，点击右上角的 + 号 → 新建仓库，填写好 仓库名称 和 路径（默认和仓库名称相同）以及 仓库介绍 ，然后点击 创建，便创建好了远程仓库。

# 2）本地仓库创建 & 关联远程仓库

# Nignx 反向代理和负载均衡

该项目采用前后端分离，在前端通过请求后端api得到数据的方式，因此前端必须发送api请求。原本的方式是，可以直接请求后端暴露的接口如http://xxx:8080/xxx...，但为了提高安全和访问速度，我们在中间插入了 Nginx 。

如，前端请求apihttp://localhost/api/employee/login，请求先经过 Nginx，Nginx 根据 conf 配置策略匹配并映射到真正的后端api接口http://localhost:8080/admin/employee/login，

反向代理：将前端发送的动态请求由 Nginx 转发到后端服务器。通过该技术，可以1）提高访问速度；2）进行负载均衡；3）保证后端服务安全（不会暴露后端接口）

所谓负载均衡，就是把大量的请求按照我们指定的方式均衡的分配给集群中的每台服务器。

Nginx 反向代理的配置方式：

Nginx 负载均衡的配置方式：

总体的 Nginx 配置文件：

http {

  map $http_upgrade $connection_upgrade{
    default upgrade;
    '' close;
  }

  # webservers 对应下面的相同字段，用于后端请求的负载均衡
  upstream webservers{
    server 127.0.0.1:8080 weight=90 ;
    server 127.0.0.1:8088 weight=10 ;
  }

  server {
    listen       80;
    server_name  localhost;

    location / {
      root   html/sky;
      index  index.html index.htm;
    }

    # 反向代理,处理管理端发送的请求
    location /api/ {
      #proxy_pass   http://localhost:8080/admin/;
      proxy_pass   http://webservers/admin/;
    }

    # 反向代理,处理用户端发送的请求
    location /user/ {
      proxy_pass   http://webservers/user/;
    }

    # WebSocket
    location /ws/ {
      proxy_pass   http://webservers/ws/;
      proxy_http_version 1.1;
      proxy_read_timeout 3600s;
      proxy_set_header Upgrade $http_upgrade;
      proxy_set_header Connection "$connection_upgrade";
    }
  }
}

# Swagger & Knife4j

Swagger 可以帮助后端生成接口文档，并进行后端接口测试。使用 Swagger 你只需要按照它的规范去定义接口及接口相关的信息，就可以做到生成接口文档，以及在线接口调试页面。官网：https://swagger.io/ (opens new window)

直接使用 Swagger 会比较繁琐，所以课程中使用到一个封装了 Swagger 的框架 Knife4j 。 Knife4j 是为 Java MVC 框架集成 Swagger 生成 Api 文档的增强解决方案。

使用方式：

  1. 导入 Knife4j 的 Maven 坐标
  1. 在配置类中加入 Knife4j 相关配置
  1. 设置静态资源映射，否则接口文档页面无法访问

当启动后端服务之后，便可以通过浏览器访问到后端接口测试界面：

Swagger 常用注解

通过注解可以控制生成的接口文档，使接口文档拥有更好的可读性，常用注解如下：

注解 说明
@Api 用在类上，例如 Controller，表示对类的说明
@ApiModel 用在类上，例如 entity、DTO、VO
@ApiModelProperty 用在属性上，描述属性信息
@ApiOperation 用在方法上，例如 Controller 的方法，说明方法的用途、作用

YApi 与 Swagger

Q：通过 Swagger 就可以生成接口文档，那么我们就不需要 Yapi 了？

A：Yapi 是设计阶段使用的工具，管理和维护接口；Swagger 是在开发阶段使用的框架，帮助后端开发人员做后端的接口测试。
